## 第 5 页：ALS 算法原理（1 分钟）

**内容**：

**公式**：
$$R_{m \times n} \approx U_{m \times k} \times V_{n \times k}^T$$

**直观理解**：
- U = 用户对 k 个"隐因子"的偏好（如：动作片因子、文艺片因子）
- V = 每部电影在这 k 个因子上的得分
- 预测评分 = U × V^T 矩阵乘法

**训练过程**：
1. 随机初始化 U 和 V
2. 固定 V，求解 U（最小二乘回归）
3. 固定 U，求解 V
4. 重复步骤 2-3 直到收敛（15 轮迭代）

**并行化关键**：步骤 2 中每个用户块独立求解（只需拉取对应电影块的 V），步骤 3 同理。天然可分布式并行。

**截图建议**：ALS 矩阵分解示意图（手绘或网上找）

**说辞**："ALS 的核心是把评分矩阵分解成用户因子和电影因子两个低秩矩阵，然后交替固定一边求解另一边，每次迭代内部各 Block 可以并行计算。"

# 教师可能提问的问题（附参考答案）

## 算法原理类

### Q1：ALS 的损失函数是什么？正则化项有什么作用？
$$\min_{U,V} \sum_{(i,j) \in \Omega} (R_{ij} - U_i V_j^T)^2 + \lambda(\|U_i\|^2 + \|V_j\|^2)$$

正则化项控制模型复杂度，防止过拟合。$\lambda$ 越大，因子向量越趋近于零，模型越简单。实验中 regParam=0.1 是平衡点。

### Q2：为什么叫"交替最小二乘"？和 SGD 相比有什么优势？
交替 = 固定 U 求 V，再固定 V 求 U，轮流。每一步都是最小二乘（凸优化，有闭式解）。相比 SGD：ALS 可天然并行（每个用户/物品 Block 独立求解），SGD 需要全局更新。

### Q3：隐因子维度 k（rank）如何选择？太大或太小有什么影响？
通过交叉验证选择。rank 太小欠拟合（表达力不够），rank 太大过拟合（学到噪声）+ 计算量增大。实验中 rank=50 是经过权衡的值。

## 系统工程类

### Q4：训练-测试集怎么划分的？为什么用 randomSplit 而不是按时间切？
`ratings.randomSplit([0.8, 0.2], seed=42)` 随机划分，避免时序偏差。MovieLens 的时间戳分布不均，按时间切会导致测试集数据分布偏移。

### Q5：已评分电影为什么要从推荐中过滤？怎么实现的？
推荐系统的价值是"发现"而非"回顾"。通过 `left anti join`：推荐结果 LEFT JOIN 用户评分记录，只保留未出现在评分表里的（movieId, userId）组合。

### Q6：冷启动问题你是怎么处理的？Jaccard 相似度是什么？

评分<5 条的用户，ALS 因子向量训练不充分。改用基于内容的方案：

**步骤**：
1. 提取用户所有已评分电影的类型（genres），合并去重 → "用户口味画像"
2. 对片库中每部电影，计算其类型与用户画像的 Jaccard 相似度
3. 排除已看过的电影，按相似度降序取 Top-N

**Jaccard 相似度**：两个集合的交集大小 / 并集大小，值在 0~1 之间。

**举例**：
```
用户 A 已评分电影的类型：
  电影1: Action|Sci-Fi|Thriller → {Action, Sci-Fi, Thriller}
  电影2: Action|Adventure|Sci-Fi → {Action, Adventure, Sci-Fi}
  
用户口味画像（取并集）: {Action, Sci-Fi, Thriller, Adventure}

候选电影 C 的类型: {Action, Sci-Fi, War}
  Jaccard = 交集{Action,Sci-Fi} / 并集{Action,Sci-Fi,Thriller,Adventure,War}
          = 2/5 = 0.4

候选电影 D 的类型: {Action, Adventure, Sci-Fi, Fantasy}
  Jaccard = 交集{Action,Adventure,Sci-Fi} / 并集{Action,Sci-Fi,Thriller,Adventure,Fantasy}
          = 3/5 = 0.6

电影 D 得分更高 → 优先推荐
```

本质就是"你喜欢动作+科幻，我就找动作+科幻含量最高的电影推给你"。

### Q7：SQLite 和内存 dict 两种模式各有什么适用场景？
内存模式适合小型数据集（1M），启动一次性加载，查询 O(1)。SQLite 模式适合大型数据集（25M），启动无需全量加载，按 userId 索引查询 O(log n)，多进程可共享。

## 评估指标类

### Q8：RMSE 和 Precision/Recall 分别衡量什么？为什么需要多指标评估？
RMSE 衡量评分预测的回归精度，Precision/Recall/NDCG 衡量推荐列表的排序质量。评分预测准 ≠ 推荐列表好——用户更关心 Top-N 推荐的排序是否合理，而不在意第 57 名的电影预测分数差了多少。

### Q9：Coverage 和 Diversity 是什么？
Coverage = 被推荐到的不同电影数 / 总电影数，衡量推荐覆盖面。Diversity = 用户推荐列表内电影对之间的平均 (1 - Jaccard)，衡量推荐列表的类型多样性。两者越高说明推荐越不集中在少数热门电影上。

## 优化与扩展类

### Q10：如果现在有一个新用户，你如何给他推荐？
当前系统：新用户没有任何评分 → 无法做个性化推荐 → 退化为热门推荐（全局平均评分最高的电影）。改进方向：冷启动引导问卷（让用户选几个喜欢的类型/电影），或使用用户注册信息（年龄、性别、地区）做统计推荐。

### Q11：如何让推荐结果随用户新增的评分实时更新？
当前方案是全量重训练。改进方向：加载已有 ALS 模型 → 用新评分数据 fine-tune 几轮迭代（warm start），或采用 Lambda 架构——离线层定期全量更新，实时层（Redis + Kafka）做增量微调。

### Q12：混合推荐中 alpha 参数怎么选择？如何验证混合推荐比纯 ALS 好？
alpha 控制 ALS 分数和内容分数的权重。选择方法：在验证集上尝试不同 alpha 值（0.4~0.9），选 Coverage/Diversity 提升最大而 RMSE 下降可接受的范围。验证：对比 `--hybrid` 和默认模式的 6 项评估指标，特别是 Coverage 和 Diversity 应有明显提升。
